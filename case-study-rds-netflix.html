The Problem: A Platform Built Without a Foundation

Rapid7's security platform had grown through acquisition and parallel product development. Each product team built its own UI. By the time the problem was fully visible, the platform had:

Design–code parity

Engineers were re-implementing the same components independently, often without the same accessibility considerations, interaction behaviors, or visual language. Designers spent enormous time manually aligning patterns that should have been shared. Accessibility audits were failing across multiple product lines for the same reasons.

But the cost wasn't only internal. Users experienced a platform that felt like a collection of products, not a system. The shared mental model that makes a complex platform learnable — once you know how something works here, you know how it works everywhere — didn't exist.

"Engineering teams lacked clarity on what 'done' looked like. Designers couldn't trust that their specs would be implemented consistently. That uncertainty compounded with every new feature."

— From the pre-RDS design audit, 2021

Why Platform Foundations Matter for Developer Tools

A design system for a developer platform isn't just a component library. It's the infrastructure that determines whether product teams can move fast while building coherent experiences — or whether every team reinvents the same solutions in slightly incompatible ways.

Netflix's Developer Platform faces an analogous challenge: how do you give teams working on observability, CI/CD, deployment, and IDE integrations a shared design vocabulary that makes their independent work feel like one platform to the engineers using it? The answer requires the same architecture: a token layer, a component contract, a governance model, and a Figma–code pipeline that keeps them synchronized.

RDS was that architecture for Rapid7. The lessons from building it inform how I'd approach the same problem at Netflix scale.

Architecture: Three Layers That Scale

I designed RDS as a three-layer system where each layer is independently useful but becomes more powerful in combination. This layered architecture was critical for incremental adoption — teams could adopt Layer 1 without Layer 3 and still get real value.

1
Foundations — Tokens, Theming & Layout
Design tokens for color, spacing, typography, border radius, shadow, and motion. Light/dark theming system built into tokens from day one. Grid and layout rules. Accessibility color contrast requirements baked into the token set, not enforced separately. This layer made it possible for DVS, UPX, and every other system to speak a shared visual language without explicit coordination.
2
Components — 60+ Accessible MUI Implementations
Enterprise MUI foundation with Rapid7-specific theme overrides. Each component defined by a full spec: anatomy, states (default, hover, focus, active, disabled, error, loading), keyboard behavior, ARIA patterns, responsive rules, and implementation guidance. Figma components maintained in parity with code via Supernova token pipeline. Engineers could ship from the Figma spec without asking a designer to clarify intent.
3
Templates & Standards — Patterns That Ship
Four standard page templates (Dashboard, Listing, Detail, Form) that product teams could adopt directly, reducing per-page layout decisions to near zero. UX Standards pillars — Accessibility, Responsiveness, Usability, Governance — integrated with the Design Forum review process. Teams using RDS templates consistently passed accessibility audits without additional remediation.

The Engineering Partnership

A design system that engineers don't trust is a design system that doesn't get used. I invested heavily in making RDS feel like an engineering asset, not a design constraint:

  • Theme override model: Worked with the MUI migration lead to define a clean override architecture. Rapid7-specific tokens applied at the theme level, meaning teams got the full MUI component surface with zero visual drift from the spec.
  • Supernova token pipeline: Design token changes in Figma propagated automatically to the code library via Supernova. Designers and engineers worked from the same source of truth without manual synchronization.
  • Component API design: Participated in API reviews for each component. Where MUI's API exposed confusing patterns, we defined opinionated wrappers. Engineers shipped consistent implementations because the API guided them there.
  • Versioning and release process: Established a semantic versioning model and changelog. Teams could track what changed, what was deprecated, and what required migration work — the same expectations they had from any well-maintained engineering dependency.

"RDS feels like a first-class engineering dependency, not a design handoff artifact."

— Senior engineer, post-adoption feedback

The Tradeoff We Accepted

What We Accepted

We chose MUI as the engineering foundation despite some product teams' preference for custom implementations. This meant accepting MUI's API constraints and opinionated behaviors in some areas, and writing more sophisticated theme overrides than a ground-up custom system would require.

Why

A custom component library built from scratch would have taken 18–24 months to reach the coverage and accessibility quality MUI already provided. The speed-to-quality tradeoff was decisive. MUI's accessibility implementations — keyboard models, ARIA patterns, focus management — represented years of community investment that we could build on rather than rebuild. Custom where it mattered; borrowed where it didn't.

Governance: Making Adoption Self-Sustaining

Systems without governance drift. I built the governance model alongside the system:

  • Design Forum integration: Every new pattern or component addition went through the Design Forum before entering the system. This prevented the proliferation of one-off additions that would re-fragment the library over time.
  • FOG evaluation model: Proposals were evaluated against Facts (research, data, accessibility requirements), Opinions (design expertise, team preferences), and Guesses (hypotheses requiring validation). This made decision-making transparent and evidence-grounded.
  • Quarterly audits: Cross-product audits measured adoption rate, identified drift, and surfaced components that needed evolution. The audit data drove the roadmap.
  • Exception tracking: Documented exceptions with rationale, allowing for genuine edge cases while maintaining visibility into where the system was being bypassed and why.
